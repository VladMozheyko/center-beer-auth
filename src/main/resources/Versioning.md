# Правила версирования приложения
### Структура

Запись: MAJOR.MINOR.PATCH

Пример: 1.4.2

- MAJOR — мажорная версия
- MINOR — минорная версия
- PATCH — патч‑версия

### Когда менять что

1. <b style=color:green>MAJOR (X.0.0)</b>

Увеличиваем, когда есть ломающие изменения (breaking changes):

- меняется контракт API так, что старый клиент перестаёт работать;
- удаляются/меняются существующие эндпоинты;
- миграции БД несовместимы назад (нельзя просто откатить код без отката БД);
- меняется логика так, что старое поведение больше не гарантируется.

Примеры:

0.5.3 → 1.0.0 — ввели новую схему авторизации, старые токены больше не подходят.

1.2.0 → 2.0.0 — удалили старый эндпоинт /oauth2/vk и оставили только /oauth2/vk_pkce.

2. <b style=color:green>MINOR (X.Y.0)</b>
   
Увеличиваем, когда добавляем новый функционал, но ничего не ломаем:

- новые эндпоинты;
- новые поля в ответах API (старым клиентам не мешают);
- новые настройки/фичи, которые по умолчанию отключены или работают совместимо.

При этом:
- MINOR увеличиваем,
- PATCH обнуляем.

Примеры:

1.2.3 → 1.3.0 — добавлен новый эндпоинт GET /api/v1/auth/sessions.

1.3.4 → 1.4.0 — добавили поддержку ещё одного OAuth‑провайдера, старые флоу не поломаны.
3. <b style=color:green>PATCH (X.Y.Z)</b>
   
Увеличиваем, когда делаем мелкие изменения, не меняющие контракт:

- багфиксы;
- оптимизации;
- правки логов;
- незначительные правки поведения, не ломающие сценарии использования.

Примеры:

1.3.0 → 1.3.1 — пофиксили баг с истечением токена.

1.3.1 → 1.3.2 — улучшили логи, поправили опечатки, поправили NPE.
